# External Code Review

## Findings

- high - `src/mmm/model.jl:206`, `src/mmm/model.jl:223`, `src/mmm/trend.jl:74`, `src/mmm/trend.jl:89`, `src/mmm/trend.jl:113`, `src/pipeline/stages.jl:128`, `/home/user/Documents/GITHUB/tandpds/abacus/abacus/mmm/additive_effect.py:467` - Posterior prediction and blocked holdout validation recompute trend features from the prediction window instead of the fitted trend basis. `_predict_time_series_mmm` rebuilds runtime from `artifact.spec` and `new_data`, but `_trend_features` normalizes dates against `first(new_data.dates)` and `maximum(new_data.dates .- first(new_data.dates))`; a future holdout therefore starts at trend value `0.0` again instead of continuing the fitted trend. Abacus stores the first fitted date and updates new data relative to that origin. This matters because any model using `trend.type = "linear"` or `"changepoint"` produces miscalibrated posterior predictions, holdout residuals, and validation metrics for future or sliced data. The shortest credible fix is to persist fitted trend state in `MMMModelSpec` or `spec.trend` and make prediction/post-model replay use the fitted origin, scale, and changepoint locations rather than deriving them from `new_data`.

- high - `src/mmm/model.jl:211`, `src/mmm/model.jl:223`, `src/mmm/holidays.jl:113`, `src/mmm/holidays.jl:143`, `src/mmm/holidays.jl:158`, `src/mmm/holidays.jl:259`, `src/pipeline/stages.jl:128` - Holiday exposure is also recomputed from the prediction data rather than a fitted calendar-period contract. `_holiday_design_matrix(spec.holidays, new_data)` infers period length from only `new_data.dates`; for a one-row holdout it defaults to one day, and for short or irregular holdouts it can use a different exposure window than the fitted model. This matters because the Phase 12 holiday-bearing row is documented as a coherent Epsilon-native design, but posterior predictions and blocked validation can silently score a different holiday exposure basis from the fitted model. The shortest credible fix is to persist the inferred observation period or date-frequency state at fit/build time and reuse it for prediction, validation, and post-model replay.

- medium - `src/transforms/saturation.jl:69`, `src/transforms/saturation.jl:80`, `src/model/types.jl:396`, `src/pipeline/stages.jl:789` - The public saturation/model path does not consistently reject negative media inputs before applying transforms that assume nonnegative spend. `hill_function` computes `x_array .^ slope_array` without validating `x >= 0`, so negative channel values with non-integer slopes can raise a raw `DomainError`; `MMMData` validates only finiteness, and the pipeline validates only positive total observed channel spend, so row-level negative spend can reach the model. This matters because users get backend/math failures instead of a clear data-contract error, and response/adstock/optimization surfaces already assume nonnegative spend. The shortest credible fix is to validate media/channel inputs as nonnegative at `MMMData` and pipeline loading boundaries, or explicitly support signed media by adding transform-level validation and documented behavior.

- medium - `src/pipeline/config.jl:203`, `src/pipeline/config.jl:249`, `src/pipeline/config.jl:348`, `src/model/config.jl:234` - The pipeline configuration accepts unknown top-level YAML keys silently. `_validate_pipeline_top_level_contract` blocks only VI keys and separate `x_path` / `y_path`; after runner keys are stripped, `model_config_from_dict` stores any remaining unknown top-level keys in `ModelConfig.extras`. This means common typos such as `optimisation:` instead of `optimization:` or `validaton:` instead of `validation:` are ignored and can disable intended stages without warning. This matters because the closed Phase 9 surface is presented as a bounded YAML contract, and silent ignores weaken reproducibility. The shortest credible fix is to reject or warn on unknown top-level keys in `run_pipeline`, with an allowlist for intentional `extras` if that is still required for the lower-level API.

## Residual Risks

- Panel `nobs(data::PanelMMMData)` currently means total target cells rather than time rows, while coordinate metadata uses `"time"` rows. That is acceptable for the current explicitly unsupported panel post-model surface, but it will become risky if panel diagnostics, summaries, or predictive tables start consuming `spec.nobs` generically.

- MCMC, VI, validation, and pipeline tests use small draw counts for runtime reasons. They exercise contracts, but they do not provide strong statistical regression protection for convergence-sensitive behavior.

- Serialized model/results artifacts require exact Epsilon and Julia version matches on load. That is strict and reproducible, but it may be too brittle for release-era artifact reuse unless documented as an intentional compatibility policy.

- This review used static inspection plus targeted Julia probes; it did not rerun the full `make test` and `make docs` quality gate.

## Conclusion

The core transform and typed artifact layers are reasonably structured, but the current prediction path has state-reconstruction gaps for date-derived features. Trend and holiday basis state should be repaired before relying on holdout validation, release parity claims, or downstream optimization/reporting generated from future-date predictions.
