# External Planning Review

## Findings

- **high** — `.planning/REQUIREMENTS.md:18-19,96-97` — `TRANS-01` and `PRIOR-01` are marked as `[ ]` (pending) with status "Pending" in the traceability table, but the codebase has fully implemented and tested all transforms (Phase 2 complete) and all prior/distribution types (Phase 3 complete). The ROADMAP, GSD-BOARD, and MILESTONES all mark these phases as done. This stale status means the requirements document — the canonical traceability source — claims two completed deliverables are still unmet. A contributor or auditor reading only REQUIREMENTS.md would conclude the project is further behind than it is. Fix by marking `TRANS-01` and `PRIOR-01` as `[x]` and updating their traceability status to "Completed."

- **high** — `.planning/MILESTONES.md:88` — M4 acceptance criterion reads: "Can load YAML config, build model, run NUTS, get posterior chains. Chains mix well (R-hat < 1.05)." However, M4 is tracked as partially complete (3 of 7 deliverables checked), and the Turing.jl dependency is not yet in `Project.toml`. The acceptance criterion cannot be met until `src/mmm/model.jl` and the Turing `@model` are implemented, which are unchecked deliverables within the same milestone. The milestone should either (a) split into two milestones with separate acceptance criteria — one for the typed config/builder layer (already done) and one for the Turing model layer (not started) — or (b) revise the acceptance criterion to reflect the incremental deliverables that are actually being verified. The current wording makes it impossible to partially accept M4, which encourages either premature "completion" or indefinite "in progress" status.

- **high** — `.planning/RISKS-AND-DECISIONS.md:18` — Risk R1 (HSGP) mitigation says "Spike early (Phase 4, week 1)." The ROADMAP places HSGP in Phase 5, plan 05-04. STATE.md lists HSGP as a blocker that "should be forced early in Phase 5, not discovered late." These three documents disagree on whether the HSGP spike belongs in Phase 4 or Phase 5. Since Phase 4 is currently in progress, this ambiguity directly affects execution: a developer could reasonably start the HSGP spike now or defer it, and either choice would be "following the plan." Fix by aligning all three documents on a single phase assignment. If the intent is to spike in Phase 4, update ROADMAP and STATE.md. If Phase 5, update RISKS-AND-DECISIONS.

- **medium** — `.planning/RISKS-AND-DECISIONS.md:46-57` — Risk R3 (Autodiff Compatibility) mitigation specifies: "Test each transform with `ForwardDiff.gradient` and `ReverseDiff.gradient` immediately after implementation." All transforms are now implemented and parity-tested, but no autodiff compatibility tests exist in the test suite. The risk document's own mitigation has not been executed. Since transforms are the foundation that every downstream model layer will depend on through Turing's autodiff, an undetected autodiff incompatibility will block the entire Phase 4 Turing model work. Fix by either (a) adding autodiff smoke tests to the transform test suite before starting the Turing `@model`, or (b) updating R3's status from "🔴 Not started" to "🟡 Partially mitigated" with a note that parity tests pass but autodiff tests are pending, and adding the autodiff test task to the Phase 4 plan.

- **medium** — `.planning/GSD-BOARD.md:100-106` — The abstract type hierarchy shown is `AbstractModelBuilder → AbstractRegressionModel → AbstractMMMModel → PanelMMM`. The actual codebase (`src/model/types.jl`) implements `AbstractModel → AbstractRegressionModel → AbstractMMMModel`, with no `AbstractModelBuilder` type. The top-level type is `AbstractModel`, not `AbstractModelBuilder`. This mismatch means a contributor following the GSD-BOARD would look for a type that doesn't exist. Fix by updating the GSD-BOARD hierarchy to match the implemented types.

- **medium** — `.planning/ARCHITECTURE.md:35` — The module structure lists `src/distributions/registry.jl` ("Prior registry (type dispatch)") as a planned file. This file does not exist and was never implemented. The GSD-BOARD Phase 3 section includes a checked item "Implement `SpecialPriorRegistry` pattern in Julia (type dispatch)," but the actual implementation uses `deserialize_special_prior` with an if/elseif chain in `special.jl`, not a separate registry module. The architecture document misrepresents the implementation structure. Fix by either (a) removing `registry.jl` from the architecture tree and noting that type dispatch is handled inline in `special.jl`, or (b) creating the registry module as planned. Option (a) is lower risk since the current approach works.

- **medium** — `.planning/ARCHITECTURE.md:32` — The module structure lists `src/distributions/special.jl` with description "Scaled, SkewStudentT, Michaelis dist." The codebase does not contain a Michaelis distribution, and the Phase 3 decision (recorded in STATE.md) explicitly closed this scope: "Do not invent unsupported Abacus custom distributions when the upstream code only exposes a transform or helper concept" and "Close Michaelis scope at the saturation layer rather than inventing a separate prior/distribution type." The architecture document is stale on this point. Fix by updating the description to "Scaled, SkewStudentT, special prior compatibility" to match the actual file contents.

- **medium** — `.planning/DEPENDENCIES.md:17-18` — Lists `DataFrames.jl` and `CSV.jl` as core dependencies "Required From Phase 0." The actual `Project.toml` has neither dependency. The codebase currently uses plain `AbstractVector`/`AbstractMatrix` types in `MMMData` rather than DataFrames. If DataFrames is truly required from Phase 0, it should be in `Project.toml`; if it is deferred to a later phase, the dependency table should reflect that. Fix by either adding DataFrames.jl/CSV.jl to `Project.toml` with compat bounds, or updating DEPENDENCIES.md to show them as "Required From Phase 4" (when data loading from files will need them).

- **medium** — `.planning/DEPENDENCIES.md:13` — Lists `Turing.jl` as "Required From Phase 3." Phase 3 is complete and Turing.jl is not in `Project.toml`. The ROADMAP places the Turing `@model` in Phase 4, plan 04-03. The dependency table's phase assignment is inconsistent with both the actual dependency state and the ROADMAP. Fix by updating Turing.jl's "Required From" to "Phase 4."

- **low** — `.planning/DEPENDENCIES.md` — Uses "Phase 0" as a phase number in multiple places (DataFrames, CSV, Dates, Test, Runic, Documenter, Aqua). The ROADMAP numbers phases 1 through 11. "Phase 0" is never defined in any planning document. Fix by mapping "Phase 0" references to "Phase 1" (Foundation), which is where these dependencies were actually introduced.

- **low** — `.planning/COMPONENT-MAPPING.md:67` — Lists "Michaelis(Km, S)" as a custom distribution needing implementation: "Implement as custom `Distribution` subtype." A contributor reading the component mapping would attempt to implement a Michaelis distribution that the project has explicitly decided not to build. - **low** — `.planning/STATE.md:57-58` — Lists "Implement config merging (defaults plus user overrides) on top of the typed loaders" as a pending todo. This same item appears as unchecked in GSD-BOARD Phase 4 (3a) and MILESTONES M4. However, no planning document specifies what "config merging" means concretely: what defaults exist, what the override precedence is, or how conflicts are resolved. The todo is too vague to execute without additional specification. Fix by adding a brief spec to the Phase 4 plan: define the default config structure, the merge semantics (deep vs shallow), and at least one example of a default being overridden.

## Cross-Document Gaps

- **REQUIREMENTS.md vs ROADMAP.md vs GSD-BOARD vs MILESTONES**: REQUIREMENTS.md marks TRANS-01 and PRIOR-01 as pending; all other documents mark Phases 2 and 3 as complete. REQUIREMENTS.md is the canonical traceability source but is the least up-to-date document.

- **RISKS-AND-DECISIONS.md vs ROADMAP.md vs STATE.md**: The HSGP spike is assigned to Phase 4 (RISKS), Phase 5 plan 05-04 (ROADMAP), and "early in Phase 5" (STATE). No two documents agree.

- **DEPENDENCIES.md vs Project.toml**: DEPENDENCIES.md lists 11 core dependencies; Project.toml has 2 (Distributions.jl, YAML.jl). The gap between planned and actual dependencies is large and unacknowledged. Most critically, Turing.jl is listed as "Required From Phase 3" but Phase 3 completed without it.

- **ARCHITECTURE.md vs codebase**: The architecture tree lists `registry.jl` (not implemented) and describes `special.jl` as containing "Michaelis dist" (not implemented, explicitly decided against). The GSD-BOARD shows `AbstractModelBuilder` (not in codebase). These are not future plans — they are stale descriptions of what was supposedly already built.

- **GSD-BOARD Phase 3 vs actual tests**: The GSD-BOARD claims "Implement `SpecialPriorRegistry` pattern in Julia (type dispatch)" is complete, but no registry module or dispatch-based registry exists. The actual implementation uses an if/elseif chain. The checked item overstates what was delivered.

## Recommended Planning Changes

1. **Update REQUIREMENTS.md**: Mark TRANS-01 and PRIOR-01 as `[x]` completed. Update the traceability table status to "Completed." This is the single most important stale-state fix because REQUIREMENTS.md is the canonical source for whether the project meets its stated goals.

2. **Split M4 acceptance criteria**: Either split M4 into two milestones (M4a: typed config/builder, M4b: Turing model) or rewrite the acceptance criterion to be incremental: "M4 partial: Can load YAML config and build a typed model specification. M4 full: Can run NUTS and obtain well-mixed posterior chains."

3. **Align HSGP spike phase assignment**: Pick one phase (4 or 5) and update all three documents (RISKS-AND-DECISIONS, ROADMAP, STATE) consistently. If the spike stays in Phase 5, change RISKS-AND-DECISIONS line 18 from "Phase 4, week 1" to "Phase 5, plan 05-04."

4. **Execute or defer R3 autodiff mitigation**: Add autodiff smoke tests to the transform test suite before Phase 4 Turing work begins, or update R3 status to reflect that the mitigation is deferred. Do not leave R3 in "Not started" state while the code it protects is already landed.

5. **Fix GSD-BOARD type hierarchy**: Change `AbstractModelBuilder` to `AbstractModel` to match the codebase.

6. **Update ARCHITECTURE.md module tree**: Remove `registry.jl`, update `special.jl` description to remove "Michaelis dist," and mark future files like `io.jl` as `(planned)`.

7. **Update DEPENDENCIES.md phase numbers**: Change "Phase 0" to "Phase 1," change Turing.jl "Required From" to "Phase 4," and add a note that DataFrames.jl/CSV.jl are not yet in Project.toml.

8. **Update COMPONENT-MAPPING.md Michaelis entry**: Add a note that Michaelis-Menten is implemented as a saturation transform only, with no separate distribution type.

9. **Specify config merging**: Add a brief spec (3-5 lines) to the Phase 4 plan or STATE.md pending todos defining what "config merging" means: default structure, merge semantics, and one example.

## Conclusion

The planning set is well-organized and internally rich, but it has drifted from the codebase in specific ways that matter for execution. The most urgent problem is stale status: REQUIREMENTS.md claims two completed phases are still pending, which undermines traceability. The second problem is the M4 acceptance criterion, which cannot be incrementally satisfied — this will cause either premature closure or indefinite limbo. The third problem is the HSGP spike phase disagreement, which creates immediate ambiguity for the current Phase 4 work. These three issues should be resolved before the next execution sprint. The remaining findings are documentation accuracy problems that are less time-sensitive but should be corrected to prevent contributors from following stale plans.
