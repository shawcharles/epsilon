# External Code Review

## Findings

- [high] `src/optimization/optimizer.jl` (`_project_to_constraint_bounds`, `_solve_budget_optimization_problem`), `src/optimization/objective.jl` (`_allocation_vector`) — post-solve bound snapping can break the fixed-budget equality contract. `_project_to_constraint_bounds` snaps each near-bound coordinate independently, but only attempts to rebalance if the residual is already within the same tolerance; when several coordinates snap at once, it can return an allocation whose sum no longer equals `total_budget`. `_solve_budget_optimization_problem` then feeds that allocation into `_evaluate_budget_objective`, which explicitly requires exact fixed-budget equality. This can turn a solver-success path into a late failure or emit an allocation that violates the public contract. The shortest credible fix is to re-project onto the equality constraint after snapping, or avoid snapping unless the equality can still be preserved, and add a regression test with multiple near-bound channels.

- [high] `src/model/types.jl` (`validate_mmm_data`, `validate_panel_mmm_data`, `_validate_numeric_values`) — typed dataset validation accepts `NaN`, `Inf`, and zero-row inputs. The current checks only verify shape alignment and `value isa Real`, so obviously invalid data passes public constructors and fails later inside model building, summaries, or Turing with much worse diagnostics. This matters because the typed data layer is the repository’s main programmatic API boundary. Add explicit non-empty checks plus finite-value validation for `target`, `channels`, `controls`, and `events`, and cover `NaN`/`Inf`/empty cases in `test/model/types.jl`.

- [medium] `src/transforms/scaling.jl` (`fit!`, `transform`, `inverse_transform` for `MaxAbsScaler` and `StandardScaler`) — fitted scalers do not record or validate the feature shape they were fit on. A scaler fit on a matrix can later transform a vector and silently use only the first column’s statistics, while a scaler fit on a vector can be broadcast across every column of a matrix. That is silent numerical corruption in a public preprocessing API, not just an ergonomic issue. Store the fitted rank/feature count and reject shape-mismatched `transform` and `inverse_transform` calls; add negative tests for vector↔matrix misuse.

- [medium] `src/model/types.jl` (`ModelConfig`, `validate_model_config`), `src/postmodel/metrics.jl` (`_default_metric_name`), `src/optimization/optimizer.jl` (`_default_efficiency`) — `target_type` is under-validated, but downstream behavior depends on it. `ModelConfig` accepts arbitrary strings, while post-model and optimization defaults switch to CPA semantics only for the exact lowercase string `"conversion"`. Inputs like `"conversions"` are accepted and then silently fall back to revenue/ROAS behavior. This is a broken public API assumption with real analyst impact, because default efficiency reporting changes without any explicit error. Validate `target_type` against the supported vocabulary (or normalize accepted synonyms) at config construction, and add end-to-end tests for conversion-target behavior.

- [medium] `src/transforms/adstock.jl` (`weibull_adstock`, `_parse_weibull_type`), `src/Epsilon.jl` export surface, `test/transforms/adstock.jl` (`type parsing`) — the public `weibull_adstock` docstring advertises `type=PDF` / `type=CDF`, but `PDF` and `CDF` are not exported by `using Epsilon`. A documented top-level call like `weibull_adstock(x; type=PDF)` fails with `UndefVarError`. This matters because it makes the public API documentation non-executable by default. Either export and document those enum values, or rewrite the docstring/examples to use the actually supported spellings such as `Epsilon.PDF`, `:pdf`, or `"pdf"`.

- [low] `src/model/types.jl` (`_string_vector`, `_string_tuple`, direct constructors for `ModelConfig`, `MMMData`, `PanelMMMData`) — direct typed constructors fail unclearly when users pass a scalar string where a string collection is required. Instead of an argument-validation error, the code falls through internal iteration and raises `MethodError(String(::Char))`. That makes direct-constructor misuse much harder to diagnose than the YAML path, which already checks list types explicitly. Reject `AbstractString` inputs for vector/tuple fields with an `ArgumentError`, and add direct-constructor regression tests.

- [low] `src/transforms/scaling.jl` (`validate_channel_values`) — this exported helper indexes `data[:, channel_columns]` without validating the indices first, so invalid column selections raise a raw `BoundsError` instead of the stable `ArgumentError` contract used by `validate_column_indices`. Public validation helpers should not depend on an undocumented call order. Validate or normalize `channel_columns` inside `validate_channel_values`, or document the precondition explicitly and test it.

- [low] `src/transforms/adstock.jl` (`_binomial_adstock_weights`, `binomial_adstock`) — binomial adstock widens `Float32` inputs to `Float64` because its kernel is built from `Float64` literals and `collect`-based arithmetic, unlike the other adstock variants that preserve `Float32`. This matters for type stability, allocations, and consistency in a numerically sensitive transform layer that advertises generic array support. Build the kernel in a promoted float type derived from `x`/`alpha`, and add an eltype regression test.

## Residual Risks

- Conversion-target workflows still carry disproportionate risk because the current test suite is dominated by `target_type = "revenue"` cases; even after fixing the config hole, the CPA-specific path needs stronger positive-path integration coverage.
- Pipeline coverage is mostly small happy-path runs on a single fixture dataset. The stage contract is exercised, but broader schema compatibility, error recovery, and real-world malformed-input cases remain relatively shallow.
- `PanelMMM` is intentionally bounded and explicitly excluded from post-model, optimization, and pipeline layers. That is an honest current contract, but it means any later expansion into those surfaces will be high risk because downstream coverage is still thin.
- `.planning/STATE.md` still flags PyMC-to-Turing parameterization differences as a blocker. Current tests cover transforms and selected downstream fixtures, not a full end-to-end parity suite across the supported modeling surface.

## Conclusion

Current code health is mixed. The repository has substantial structure and tests, but several public APIs still accept invalid inputs or silently return wrong results, and the optimizer has a concrete fixed-budget consistency bug on a nominal success path. Passing tests are not yet strong evidence of robustness for invalid data, conversion-target semantics, or near-bound optimization cases.
