# External Code Review

## Findings

- **high** — `src/distributions/special.jl:245` — `SkewStudentT` computes `logpdf` as `log(pdf(d, x))`. When `pdf` returns a value near zero, taking the log discards most of the floating-point range and can return `-Inf` where the true log-density is finite. This matters because `logpdf` is the primary hook used by probabilistic programming systems (Turing, DynamicHMC) for density evaluation. A future Turing model using `SkewStudentT` will silently produce wrong posterior geometry or hard-to-diagnose numerical errors. Fix by implementing `logpdf` directly from the Azzalini-Capitanio formula: `log(2) - log(σ) + logpdf(TDist(ν), z) + logcdf(TDist(ν + 1), α * z * sqrt((ν + 1) / (ν + z²)))`, where `z = (x - μ) / σ`.

- **high** — `src/transforms/adstock.jl:299-301` — Weibull PDF adstock normalizes weights via `(weights .- min_weights) ./ (max_weights .- min_weights)`. When all weights along the last axis are equal (degenerate but valid input, e.g. `l_max = 1`), `max_weights - min_weights` is zero and the result is `NaN`. The `l_max > 0` guard in the public function allows `l_max = 1`, which produces a single weight and triggers this division-by-zero. Any downstream consumer receiving `NaN` weights will propagate NaN through the convolution silently. Fix by guarding: where `max ≈ min`, return `zero(weights)` (or `fill(1/size, ...)` if uniform weighting is preferred).

- **high** — `src/transforms/saturation.jl:69-79` — `hill_function` and `michaelis_menten` perform no parameter validation, unlike every other transform in the codebase (`logistic_saturation` validates `lam ≥ 0`, `tanh_saturation` validates `b > 0` and `c > 0`, all adstock functions validate `alpha` ranges). Negative `slope` or `kappa` in `hill_function` and non-positive `alpha` or `lam` in `michaelis_menten` will silently produce wrong outputs (e.g. `kappa^slope` with negative `slope` returns a complex number for negative `kappa`, which will error or produce `NaN` depending on the path). Fix by adding `_validate_positive` / `_validate_nonnegative` calls matching the pattern used in `adstock.jl` and `saturation.jl` for the other functions.

- **medium** — `test/distributions/priors.jl:101-124` — `instantiate_distribution` is tested for only 4 of its 14 handled distribution types (`Normal`, `Gamma`, `Exponential`, `TruncatedNormal`). The following are untested: `HalfNormal`, `Beta`, `Uniform`, `Weibull`, `Cauchy`, `HalfCauchy`, `StudentT`, `Laplace`, `LogNormal`. A parameterization mistake in any of these paths (e.g. the `Gamma` rate-vs-shape flip that is already correctly handled) would go undetected. The `Gamma` path was tricky enough to warrant careful testing (it converts `beta` rate to `1/beta` for Distributions.jl's shape-scale parameterization); the other paths deserve the same scrutiny. Fix by adding at least one `instantiate_distribution` + property check per untested distribution type.

- **medium** — `src/Epsilon.jl:13-14,66-67` — `fit!` and `predict` are exported public API but both throw errors at runtime (`:unimplemented` backend, `ErrorException`). Users who see these in autocomplete or docs will reasonably assume they work. This violates the project standard that "internal APIs stay unexported until intentionally promoted." Fix by either (a) unexporting them until the Turing backend lands and re-exporting at that point, or (b) adding a docstring that explicitly states the function is a placeholder and will error, so at least `?fit!` tells the truth.

- **medium** — `src/distributions/special.jl:84-89` — `LogNormalPrior(; mu=..., sigma=..., mean=..., std=...)` silently reassigns: if `mu` is provided without `mean`, it renames the key to `mean`; same for `sigma` → `std`. If a user provides both `mu` and `mean`, `mean` wins and `mu` is silently dropped. This is a config-compatibility hazard: a user writing `LogNormalPrior(; mu=4.0, mean=5.0)` gets `mean=5.0` with no warning. Fix by throwing an `ArgumentError` when both aliases are provided, matching the pattern used in `_read_distribution_name` for `dist`/`distribution`.

- **medium** — `src/distributions/masked.jl:84-87` — `_is_masked_prior_mapping` returns `true` for any dict containing both `:prior` and `:mask` keys, regardless of structure. A user config dict that happens to use those key names for unrelated data will be misidentified as a `MaskedPrior` and either deserialized incorrectly or throw a confusing error. Fix by tightening the heuristic: require `:class == "MaskedPrior"` OR (`:prior` AND `:mask` AND `:prior` is itself a prior-like mapping).

- **medium** — `src/transforms/scaling.jl:88,99` — `StandardScaler` computes standard deviation with `sum(.^2) / length` (population std, dividing by N). This matches sklearn's `StandardScaler` but may not match Abacus/PyMC, which typically uses sample std (dividing by N-1). If Abacus preprocesses data with `ddof=1`, Epsilon will produce systematically different scaled values, breaking parity for any test that compares preprocessed data rather than final model outputs. Fix by verifying the Abacus preprocessing path's `ddof` setting and, if it uses 1, adding a `ddof` parameter to `StandardScaler` defaulting to the matching value.

- **low** — `test/distributions/priors.jl` — There is no round-trip test for `EpsilonPrior` via `to_dict` / `deserialize_prior`. The shrinkage and special priors each test their own round-trip, but the base `EpsilonPrior` serialization path (including `dims`, `centered`, `transform` fields) is untested. A serialization bug in the base prior would only be caught indirectly. Fix by adding a test: `deserialize_prior(to_dict(prior)) == prior` for an `EpsilonPrior` with non-default `dims`, `centered`, and `transform`.

- **low** — `src/distributions/masked.jl:8` — `MaskedPrior.prior` is typed as `Any`. This forces the compiler to emit dynamic dispatch at every call site that reads `prior.prior`, creating a type instability in any hot path that iterates over masked priors (e.g. a future Turing model that loops over channel priors). Fix by parameterizing: `MaskedPrior{P} prior::P` where `P` can be `EpsilonPrior`, `AbstractSpecialPrior`, etc. This is a minor performance concern now but will matter once the Turing model layer iterates over priors in the inner loop.

- **low** — `src/model/types.jl:224-232` — `_string_key_dict` performs a shallow `copy` for `Dict{String, Any}` inputs. Nested dicts inside the copied dict share references with the original. If a caller later mutates a nested dict in the copy, it also mutates the original. No current code path does this, but the function's contract implies isolation. Fix by using `deepcopy` or documenting that the copy is shallow.

## Open Questions

- The Weibull CDF adstock branch (`src/transforms/adstock.jl:302-307`) computes `cumprod` of a prepended-ones survival vector, which produces values that are NOT the Weibull CDF despite the naming. The parity tests pass, so this matches Abacus behavior. Is the "CDF" label a historical name from PyMC-Marketing that actually refers to a different kernel shape? If so, a brief comment in the code explaining the naming would prevent future contributors from "fixing" it to match the statistical CDF.

- `StandardScaler` uses population std (N denominator). Does Abacus use sample std (N-1) in its preprocessing? If the parity tests only compare final model outputs, a systematic scaling difference could be masked by downstream parameter absorption. This should be verified against the Abacus source.

## Residual Risks

- The `SkewStudentT` distribution has no analytical `quantile`/`rand` method that is verified against a reference implementation. The current `rand` uses a stochastic representation via `SkewNormal` + chi-squared, which is correct in principle but untested for distributional correctness (only finiteness is checked). Once Turing models use this distribution, any subtle parameterization error will be hard to diagnose from MCMC output alone.

- The config deserialization layer (`deserialize_prior`, `deserialize_model_config`, `model_config_from_dict`) has significant heuristic dispatch logic (`_is_prior_mapping`, `_is_special_prior_mapping`, `_is_masked_prior_mapping`). These heuristics are tested on well-formed inputs but not on adversarial or ambiguous inputs. As the config schema grows, the risk of false-positive prior identification increases.

- The `TimeSeriesMMM` mutable struct is the only mutable model object in the codebase. Its `built_model` and `fit_state` fields are mutated by `build_model` and `fit!` respectively. Once concurrent or nested model fitting is attempted, this shared mutable state will be a concurrency hazard.

- No tests exercise the full YAML config loading path with invalid or partial YAML files (e.g., missing required sections, wrong types for scalar fields, empty media section). The `model_config_from_dict` tests cover the dict-level path, but `YAML.load_file` can produce surprising types (e.g., `Int` vs `Float64`, empty `Dict` vs `Nothing`) that the dict-level tests may not reproduce.

## Conclusion

The codebase is well-structured for its current phase: clean module layout, consistent naming, real parity fixtures, and honest stub markers. The main risks are numerical rather than architectural. The `log(pdf(...))` implementation in `SkewStudentT` will cause real problems the moment a Turing model tries to use it for density evaluation, and the Weibull normalization edge case will produce silent `NaN` under valid input. Both are cheap to fix now and expensive to debug later. The missing parameter validation in `hill_function` and `michaelis_menten` is an inconsistency that should be closed before the API stabilizes. The test gaps in `instantiate_distribution` and `EpsilonPrior` serialization are the most important coverage holes — they protect the parameterization translation layer that sits between user config and the statistical runtime.
