# External Code Review

## Findings

- **high** — `src/mmm/model.jl:81-83`, `src/model/builder.jl:111-135,172-183`, `test/model/builder.jl:182-227` — `fit!` reuses `model.built_model` without rebuilding or revalidating a mutable `TimeSeriesMMM`, so mutating `model.data` or `model.config` after `build_model(model)` can silently fit one dataset while persisting a stale `artifact.spec`. This makes downstream diagnostics, saved artifacts, and any code that trusts `spec.nchannels`/`spec.nobs` unreliable; `fit!` should rebuild or at least rerun alignment validation before sampling.

- **high** — `src/model/results.jl:54-81`, `test/model/results.jl:40-71` — `model_results(model; new_data=...)` generates posterior/prior predictive draws for `new_data` but always returns the training `artifact.spec`. This produces internally inconsistent `ModelResults` objects where `results.spec.nobs` can disagree with the predictive variables actually stored in `posterior_predictive`/`prior_predictive`, which will mislead downstream diagnostics/export code; either rebuild a spec for `new_data` or reject out-of-sample `new_data` here.

- **high** — `src/transforms/adstock.jl:297-300`, `test/transforms/adstock.jl:83-123` — the Weibull CDF adstock branch constructs an effective kernel of length `l_max + 1` by prepending a leading `1` and returning `cumprod` over the padded array. The public API still describes `l_max` as the lag limit, yet `weibull_adstock([1,0,0,0], 1.0, 1.0, 1; type=Epsilon.CDF)` produces carryover beyond a single lag. This changes transform semantics under valid inputs and is not covered by a boundary test; generate exactly `l_max` weights or document a different contract.

- **medium** — `src/transforms/adstock.jl:76-79,193-209,304-305`, `test/transforms/adstock.jl:58-80` — `delayed_adstock(...; normalize=true)` can return `NaN` for accepted inputs because `_validate_alpha` allows `alpha = 0`, `_delayed_adstock_weights` can then produce an all-zero kernel when `theta` lies outside `0:(l_max - 1)`, and `_normalize_last_axis` divides by the zero sum. Silent NaNs in a core transform can contaminate model inputs and posterior geometry; guard zero-sum normalization or reject parameter combinations that imply an empty kernel.

- **medium** — `src/transforms/saturation.jl:8-17`, `test/transforms/saturation.jl:18-30` — `logistic_saturation` uses `(1 - exp(-z)) / (1 + exp(-z))`, which overflows to `Inf/Inf` for large negative `z` and returns `NaN` even though the function accepts general `Real` inputs. Any caller that passes centered, differenced, or otherwise negative media inputs can get undefined outputs instead of a bounded saturation response; switch to a numerically stable formulation such as `tanh(z / 2)` or branch on sign.

- **medium** — `src/distributions/special.jl:229-236`, `src/Epsilon.jl:35`, `docs/src/index.md:113`, `test/distributions/special.jl:5-21` — `Scaled` exposes a broken public statistical interface: `minimum`/`maximum` are hardcoded to `-Inf`/`Inf` instead of reflecting the scaled base distribution, and `mean`/`var` are defined as new local generics rather than extending `Statistics.mean`/`Statistics.var`, so `mean(Scaled(...))` and `var(Scaled(...))` fail with `MethodError`. This makes an exported distribution incompatible with normal Julia statistical tooling; derive support from `base` and extend the `Statistics` methods explicitly.

- **medium** — `src/model/config.jl:134-146`, `test/model/config.jl:121-130` — config list validation assumes every vector element can be passed to `String(item)`, so inputs like `channels = [1]` throw a raw `MethodError` instead of the advertised `ModelConfigError("... must be a list of strings")`. That weakens the YAML/configuration surface and gives users unstable low-level errors for malformed input; validate element types before conversion and keep failures inside the documented config error path.

- **medium** — `src/model/types.jl:137-171,230-249`, `test/model/types.jl:56-81` — `MMMData` validates only shapes and names, not that `target`, `channels`, or `controls` are numeric. Invalid datasets such as `MMMData(target=["a", "b"], ...)` construct successfully and then fail much later inside Turing with internal errors like `MethodError(isnan, ...)`. This accepts invalid public API state and turns simple data issues into hard-to-debug sampler failures; reject non-numeric payload types at `MMMData` construction time.

- **medium** — `src/distributions/priors.jl:167-183,397-402`, `src/distributions/special.jl:147-172`, `test/distributions/special.jl:5-44`, `test/distributions/priors.jl:146-149` — nested-prior instantiation is inconsistent for `Scaled` and `SkewStudentT`. The early `Scaled`/`SkewStudentT` branches bypass the general nested-prior guard, `_resolve_distribution_parameter` only knows how to instantiate nested `EpsilonPrior`, and nested `AbstractSpecialPrior` or nested location/scale priors fall through to either a misleading `ArgumentError` or a raw `MethodError`. This makes `instantiate_distribution` unpredictable on documented public prior objects; either support nested special priors consistently or reject all nested parameters before dispatch with a targeted error.

- **low** — `src/distributions/shrinkage.jl:43-67,103-163`, `docs/src/index.md:95-96,109-112`, `test/distributions/shrinkage.jl:15-45` — several exported shrinkage-prior hyperparameters are currently dead metadata. `FinnishHorseshoePrior.slab_df` never affects `regularized_local_scales`/`finnish_horseshoe_coefficients`, and `R2D2Prior.mean_R2` plus `concentration` never affect `r2d2_variance_weights`/`r2d2_coefficients`; negative `tau2` is also not validated and falls through to a downstream `DomainError` on `sqrt`. This overstates what the current API actually implements and can mislead parity or prior-tuning work; either wire the parameters into the helper math or reject/omit unsupported knobs and validate `tau2` directly.

- **low** — `src/transforms/scaling.jl:42-59,228-235`, `test/transforms/scaling.jl:69-76` — column-index validation is inconsistent: exported `validate_column_indices` accepts non-integer indices such as `[1.5]`, while wrapper constructors like `MaxAbsScaleChannels([1.5])` fail earlier with raw `InexactError`. Callers therefore cannot rely on a stable validation path for column selectors; enforce integer indices in `validate_column_indices` and normalize constructor failures to `ArgumentError`.

- **low** — `README.md:9-11`, `docs/src/index.md:7-20`, `.planning/STATE.md:9-19` — contributor-facing status text still says the repository is in Phase 4, while the planning/state documents place the project in Phase 5. This is direct documentation drift inside the repository and can mislead contributors about current scope and completion criteria; update the README/docs status text when planning state changes.

## Residual Risks

- The Turing-backed model path is tested mostly with very small synthetic datasets and low draw counts. Serialization, diagnostics, and predictive paths work as smoke tests, but they have not been stressed under realistic chain sizes, poor mixing, or larger observation counts.

- Transform parity coverage against Abacus fixtures is strong on the main happy paths, but several boundary failures above were only exposed by targeted probes rather than committed tests. Additional edge-case tests are still needed around lag limits, normalization degeneracy, and sign/overflow behavior.

- The config/prior deserialization layer still relies heavily on `Dict{String, Any}` plus heuristic prior detection. Even where current behavior is acceptable, that loose typing leaves room for late failures as Phase 5/6 features expand the model/config surface.

## Conclusion

The repository passes its current automated suite, but several public APIs still admit invalid or misleading states that the suite does not catch. The highest near-term risks are metadata drift between what the model fit/predict paths actually do and what `MMMModelSpec`/`ModelResults` claim they did, followed by transform edge cases that return silent `NaN`s or extra carryover under valid inputs. The current tests are strongest on parity fixtures and happy-path smoke coverage, and materially weaker on boundary semantics and invalid-input handling.
